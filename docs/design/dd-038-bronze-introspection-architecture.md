# DD-035: Bronze Source Introspection & Layered dbt Architecture

**Status:** Proposed  
**Date:** 2026-06-01  
**Affects:** `integration/sources/`, `_sources.yml` generation, dataplatform repos, dbt projector  
**Implementation:** New `import-source` CLI command, `extract_source_schema` dbt macro,
modified `_sources.yml` generation in `medallion_dbt_projector.py`  
**Supersedes:** Partially evolves DD-015 (Vocabulary TTL as Bronze Contract)

---

## Context

DD-015 established that vocabulary TTL files are the authoritative bronze contract.
However, in practice:

1. **Vocabulary drift** — Actual lakehouse tables evolve (columns added/removed, type
   changes, JSON payloads) and the vocabulary TTL is updated manually, leading to
   stale contracts.
2. **Physical binding in hub** — The dbt projector currently generates `_sources.yml`
   with hard-coded `database:` and `schema:` from vocabulary metadata. This couples
   the hub to a specific environment's physical layout.
3. **No dataplatform scaffold** — Teams manually set up dbt projects for consuming
   hub projections, leading to inconsistency and missing the introspection tooling.
4. **Environment differences** — Bronze tables may differ between dev/staging/prod
   (extra columns in dev, different schemas, etc.). A single vocabulary cannot
   represent all environments without losing precision.

---

## Decision

### 1. Hybrid Introspection Pipeline

Introduce a two-stage pipeline for keeping vocabulary in sync with actual bronze:

```
┌──────────────────────────────────┐         ┌─────────────────────────────────────┐
│  DATAPLATFORM REPO               │  YAML   │  ONTOLOGY HUB                       │
│                                  │────────▶│                                     │
│  dbt run-operation               │         │  kairos-ontology import-source       │
│  extract_source_schema           │         │  → refreshes *.vocabulary.ttl        │
│  (adapter.get_columns_in_relation)│         │  → change report                    │
└──────────────────────────────────┘         └─────────────────────────────────────┘
```

**Stage 1 — Schema Extraction (dataplatform repo):**
- A dbt macro `extract_source_schema` uses dbt's built-in
  `adapter.get_columns_in_relation(source('system', 'table'))` to query the actual
  bronze tables.
- Works on ALL dbt-supported adapters (Fabric Lakehouse, Fabric Warehouse,
  Databricks, Snowflake, etc.) without platform-specific SQL.
- Outputs a standardized **YAML** intermediate file (see spec below).

**Stage 2 — Vocabulary Refresh (toolkit CLI):**
- New CLI command: `kairos-ontology import-source --from <schema.yaml> --system <name>`
- Reads YAML, compares with existing vocabulary TTL, produces an updated
  `.vocabulary.ttl` with additions, deprecations, and type changes.
- Supports `--dry-run` for preview without writing.

### 2. Layered dbt Source Separation

Split responsibility for dbt source definitions between hub and dataplatform:

| Concern | Ontology Hub (producer) | Dataplatform (consumer) |
|---------|------------------------|------------------------|
| Table/column names | ✅ From vocabulary | ✅ From introspection |
| Data types | ✅ From vocabulary | ✅ From introspection |
| Transform logic (SQL) | ✅ From SKOS + kairos-map | ❌ Consumes from hub |
| Domain semantics | ✅ Ontology classes/properties | ❌ |
| Physical database/schema | ❌ Logical refs only | ✅ `_sources.yml` |
| Connection profile | ❌ | ✅ `profiles.yml` |
| Environment config | ❌ | ✅ Dev/staging/prod targets |

**Hub's projected `_sources.yml`** declares source names and tables (for dbt graph
resolution) but does NOT specify `database:` or `schema:`. Example:

```yaml
# Generated by hub — logical source declaration only
version: 2
sources:
  - name: adminpulse
    # database: and schema: intentionally omitted — defined in dataplatform
    tables:
      - name: tblClient
      - name: tblInvoice
      - name: tblClientType
```

**Dataplatform's `_sources.yml`** provides the physical binding:

```yaml
# Dataplatform — physical source binding
version: 2
sources:
  - name: adminpulse
    database: bronze_lakehouse
    schema: raw_adminpulse
    tables:
      - name: tblClient
      - name: tblInvoice
      - name: tblClientType
```

dbt's source override mechanism resolves `{{ source('adminpulse', 'tblClient') }}`
against the dataplatform's definition when both are present in the project graph.

### 3. Dataplatform Scaffold

A new `kairos-ontology init-dataplatform` command and `kairos-setup-dataplatform`
skill scaffold a properly structured dataplatform repo with:

- dbt project configured to consume the hub via `dbt deps`
- Physical `_sources.yml` template
- `extract_source_schema` macro pre-installed
- `pyproject.toml` with uv + kairos-ontology-toolkit as dependency
- CI workflows for `dbt build` and `dbt test`

## YAML Intermediate Schema (v1.0)

The intermediate format between schema extraction and vocabulary generation:

```yaml
version: "1.0"
system: "adminpulse"                # Source system identifier (matches hub convention)
platform: "fabric-lakehouse"        # Detected from dbt adapter type
environment: "production"           # From dbt target name
extracted_at: "2026-06-01T19:00:00Z"
connection:
  database: "bronze_lakehouse"      # For dataplatform _sources.yml (not sent to hub)
  schema: "raw_adminpulse"          # For dataplatform _sources.yml (not sent to hub)

tables:
  - name: "tblClient"
    columns:
      - name: "ClientId"
        data_type: "int"            # Normalized type (dbt adapter-specific → canonical)
        nullable: false
        is_primary_key: true
      - name: "ClientName"
        data_type: "string"
        nullable: true
      - name: "MetadataJson"
        data_type: "string"
        content_type: "json-object" # Manual annotation for JSON-in-string columns
        nullable: true
    incremental_column: "ModifiedDate"  # Optional: for SCD/incremental loading

  - name: "tblInvoice"
    columns:
      - name: "InvoiceId"
        data_type: "int"
        nullable: false
        is_primary_key: true
      - name: "Amount"
        data_type: "decimal"
        nullable: true
```

### Type Normalization

The extraction macro normalizes platform-specific types to canonical names:

| Canonical Type | Fabric Lakehouse | Fabric Warehouse | Databricks |
|----------------|-----------------|------------------|------------|
| `int` | `INT`, `INTEGER` | `int`, `bigint` | `INT`, `BIGINT` |
| `string` | `STRING` | `varchar`, `nvarchar` | `STRING` |
| `decimal` | `DECIMAL` | `decimal`, `numeric` | `DECIMAL` |
| `boolean` | `BOOLEAN` | `bit` | `BOOLEAN` |
| `date` | `DATE` | `date` | `DATE` |
| `datetime` | `TIMESTAMP` | `datetime2` | `TIMESTAMP` |
| `binary` | `BINARY` | `varbinary` | `BINARY` |

```
┌─────────────────────────────────────────────────────────────────────────────────┐

│  1. kairos-ontology init-dataplatform --platform fabric --hub-repo org/my-hub    │
│  2. Configure profiles.yml with lakehouse connection                             │
│  3. dbt deps (pulls ontology-hub dbt package)                                    │
├─────────────────────────────────────────────────────────────────────────────────┤
│  INTROSPECTION CYCLE (repeatable)                                                │
├─────────────────────────────────────────────────────────────────────────────────┤
│  4. dbt run-operation extract_source_schema --args '{source: "adminpulse"}'      │
│     → outputs: extracted/adminpulse-schema.yaml                                  │
│                                                                                  │
│  5. Copy YAML to ontology-hub repo                                               │
│                                                                                  │
│  6. kairos-ontology import-source --from adminpulse-schema.yaml --system admin   │

```

## Relationship to DD-015

DD-015 established: "Vocabulary TTL is the single source of truth for bronze table
structure." This decision **preserves** that principle but adds:

1. An **automated mechanism** to keep the vocabulary current (import-source)
2. A **separation of concerns** between logical structure (hub vocabulary) and
   physical binding (dataplatform `_sources.yml`)
3. A **scaffolded dataplatform** that properly consumes hub projections

The vocabulary remains authoritative for the hub's projection pipeline. The YAML
introspection is a *refresh mechanism*, not a replacement.

---

## Rationale

### Why dbt `adapter.get_columns_in_relation()` for extraction?

- **Platform-agnostic**: One macro works on all dbt-supported databases
- **Authoritative**: Queries the actual catalog, not a stale metadata file
- **Already available**: No new dependencies needed — standard dbt adapter API
- **Consistent**: Returns structured column objects with name + type

### Why YAML as intermediate format?

- **dbt-ecosystem aligned**: YAML is the native config format for dbt
- **Human-readable**: Easy to review, diff, and version-control
- **Structured**: Supports programmatic consumption without complex parsing
- **Versionable**: Can be committed alongside the vocabulary for audit trail

### Why separate physical binding from hub?

- **Environment flexibility**: Dev/staging/prod may have different database names
- **Security**: Connection strings and physical locations stay in the secured
  dataplatform repo, not in the (potentially public) ontology hub
- **dbt best practice**: Multi-project setups use source overrides by design
- **Simpler hub**: Hub only needs to know column names and types for correct SQL

### Why not auto-refresh SKOS mappings?

- **Semantic decisions**: Mapping a new column to a domain property is a design
  choice requiring context about business meaning
- **Risk of incorrect mappings**: Auto-mapping could create incorrect
  transformations that silently corrupt data
- **Interactive workflow**: The `kairos-design-mapping` skill provides guided,
  validated mapping creation with human checkpoints

---

## Consequences

### Positive

- Bronze vocabulary stays current with minimal manual effort
- Hub projections are environment-agnostic (portable across clients/environments)
- Dataplatform teams get a standardized starting point (scaffold)
- Clear separation of concerns reduces coupling between repos
- dbt `deps` versioning continues to work for hub updates

### Negative / Trade-offs

- **Two-step refresh**: Schema extraction and vocabulary import are separate
  actions (not fully automated end-to-end)
- **Breaking change**: Existing dataplatforms consuming hub `_sources.yml` with
  database/schema will need to define their own source bindings
- **JSON detection**: `content_type` must be manually annotated — dbt adapters
  don't expose this metadata
- **PK/nullable gaps**: Some adapters don't return full constraint information
  via `get_columns_in_relation`; may need supplementary queries

### Migration Path

For existing hubs/dataplatforms:

1. Hub updates dbt projector → `_sources.yml` becomes logical-only (major version bump)
2. Dataplatforms add their own `_sources.yml` with physical binding (one-time setup)
3. Teams adopt `extract_source_schema` macro for ongoing refresh
4. Existing vocabulary TTL files remain valid — import-source can merge with them

---

## Alternatives Considered

### A. Move dbt projection entirely to dataplatform

**Rejected**: Would require the toolkit + all ontologies in the dataplatform,
losing the clean producer/consumer separation and making hub releases meaningless.

### B. Use dbt's `codegen` package for source generation

**Rejected**: `codegen` generates dbt source YAML, not ontology vocabulary TTL.
It solves a different problem (bootstrapping `_sources.yml`) and doesn't integrate
with the vocabulary/mapping pipeline.

### C. Query INFORMATION_SCHEMA directly (platform-specific macros)

**Rejected**: Would require maintaining separate SQL for each platform. dbt's
adapter layer already provides the abstraction we need.

### D. Use a live database connection in the toolkit directly

**Rejected**: Would require the toolkit to have connection profiles, which breaks
the separation of concerns. The dataplatform has the profile; the toolkit has the
ontology logic. The YAML file bridges them cleanly.
